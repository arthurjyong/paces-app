# PACES AI-Examiner MVP ŌĆö build spec (binding contract)

One screen (chat + case picker + BYOK key box) + one examiner backend function + the existing 503-case corpus (287 carousel encounters + 216 standalone library cases). A candidate picks a case, the AI plays PACES examiner/patient grounded on the case file + canonical notes, reveals findings manoeuvre-by-manoeuvre, runs the viva, and marks against the official rubric with structured output. Stack: Next.js 16 (App Router, TypeScript, Tailwind v4) + `@anthropic-ai/sdk`. Deployable to Vercel; runs locally with `npm run dev`.

**Multi-provider adapter (added 2026-07-08):** the same `@anthropic-ai/sdk` code path serves several providers whose endpoints speak the Anthropic Messages API ŌĆö Anthropic, DeepSeek, Moonshot (Kimi), MiniMax ŌĆö giving the app a budget tier ~2ŌĆō50├Ś cheaper per case depending on model (DeepSeek Ōēł 30ŌĆō50├Ś, MiniMax Ōēł 5ŌĆō8├Ś, Kimi Ōēł 2ŌĆō3├Ś). The selected MODEL determines the provider (`lib/types.ts` `MODELS` registry, client-safe); everything server-side about a provider ŌĆö baseURL, auth style (x-api-key vs Bearer), invited-access env var, and capability flags (forced tool_choice, system-as-string, thinking-off, reveal-reminder) ŌĆö lives in the fixed map `lib/providers.ts` (server-only). See "Multi-provider rules" below the invariants.

**Managed gateway path (Phase 0, added 2026-07-09):** the direction is a MANAGED tier where the operator tops up **one** place that fans out across models. That place is **Vercel AI Gateway** ŌĆö added as the `gateway` provider (baseURL `https://ai-gateway.vercel.sh`, Anthropic-Messages surface, x-api-key auth, key env `DEMO_GATEWAY_API_KEY`), fronting both offered models at **zero token markup**: `anthropic/claude-sonnet-4.6` (premium) and `deepseek/deepseek-v4-pro` (budget). `MODELS` is deliberately trimmed to just these two (Arthur, 2026-07-09: "only Sonnet 4.6 vs DeepSeek Pro at the beginning"). The four direct providers stay wired in `PROVIDER_CONFIG` for BYOK-direct and as the off-Vercel escape hatch, but are not surfaced in `MODELS`. **DEPLOYED & LIVE in production 2026-07-09** (https://paces-app.vercel.app ŌĆö prod routes both offered models through the gateway via `DEMO_GATEWAY_API_KEY`; old direct model slugs now 400). The gateway is only the *money* layer; per-user accounts / institutional gating / usage metering / self top-up are a future stateful layer (Phases 1ŌĆō2 ŌĆö binding spec: `../_MANAGED_TIER_PLAN.md`; context in `../_APP_HANDOFF.md`), NOT part of this change.

## Nomenclature (binding ŌĆö the three case axes; locked with Arthur 2026-07-06)

Every case is described by three INDEPENDENT axes. Never mix them in labels, filters, or copy:

1. **Classification** ŌĆö what kind of encounter it is: `consultation` | `communication` | `examination` (the `encounterType` field). This is the picker's default grouping and its filter chips. (Consult = old "Station 5" = brief clinical consultation ŌĆö one skill; station numbers are format artifacts and never appear on consult labels.)
2. **Theme** ŌĆö the clinical topic (`theme` field: cardiology, ophthalmology, nephrology, endocrine, ŌĆ”; curated in `../_index/case_themes.json`). An attribute, not a station. The four exam systems are both a station format and a theme. UI: tick-box multi-select with Select all.
3. **Source** ŌĆö for carousel cases: the real sitting ("NUH ┬Ę Mar 2026"). For ALL standalone cases: one pooled type-named bank ŌĆö "Consult bank" / "Communication bank" / "Examination bank" (per Arthur 2026-07-06: collection provenance is deliberately NOT user-visible; it stays internal). Source labels must never smuggle in content/topic ("Eye OSCEs" was wrong ŌĆö the eye-ness is theme=ophthalmology), author names, or initials ŌĆö and collection dir names flow into served `id`/`sitting`/`file` fields, so the DIR names must be neutral too.

`specialty` (filename-derived: Cardiovascular/ŌĆ”/Communication/Consultation) is a legacy field that mixes axes 1 and 2 ŌĆö keep it server/meta-only for exam-station grouping; don't build new UI on it.

## Canonical candidate-stem format (binding ŌĆö locked with Arthur 2026-07-07; all 503 source stems rewritten to it)

The `## Candidate stem` section of every case file is **"official exam prose"**: 2ŌĆō4 short plain-prose paragraphs in the style of the real MRCP PACES written instruction. **Near-plain text only** ŌĆö no bold, bullets, blockquotes, `---`, headings, tables. No tutor/meta commentary, no encounter-timing sentences (the card header shows authoritative timing), no station numbers, no diagnosis reveals (given diagnoses in communication scenarios are fine ŌĆö they're scenario information).

- **Consultation:** "You are the [role] in [setting]." ŌåÆ patient + quoted referral ŌåÆ "On arrival: [vitals prose]" (if given) ŌåÆ "Your task is to assess the patient's problems and address any questions or concerns raised by the patient. You will then discuss the case with the examiners."
- **Communication:** "You are the [role] in [setting]." ŌåÆ "You are about to speak to [person]." + door-given background ŌåÆ available information (if given) ŌåÆ "Your task is to [task]. You will then discuss the case with the examiners."
- **Examination (brief):** "This [age]-year-old [man/woman] [one-line context]." ŌåÆ "Please examine [system] and then present your findings, with your diagnosis and differential. Do not take a history."

Any future enrichment/generation MUST emit stems in this format ŌĆö the runtime spoiler/timing/artifact strips in `getCaseStem()` are safety nets, not the convention (as of the 2026-07-07 rewrite they strip 0 blocks corpus-wide).
**Shared types:** `lib/types.ts` (already written ŌĆö code against it, never redefine its types).

## Security invariants (any violation is a critical bug)

1. **Hidden content never reaches the client.** The client may only ever receive: the manifest (`PublicCaseMeta` fields ŌĆö `CaseMeta` minus `canonicalSlugs` and `file`; `canonicalSlugs` is matched from the case H1 title so the slugs name the diagnosis, per invariant 4's rationale), the `stem` of a case, the examiner model's chat replies, and the marksheet. The case H1 title, expected findings, model presentation, viva answers, answer key, surrogate brief, canonical notes, and assembled prompts must NEVER appear in any client-visible payload, including error messages. `displayTitle` is built from filename metadata only (the H1 title names the diagnosis ŌĆö it is a spoiler).
2. **The user's API key is never stored or logged server-side.** It arrives per-request in the `API_KEY_HEADER` header, is passed straight to the Anthropic SDK constructor (as `apiKey` or, for Bearer-auth providers, `authToken`), and is never written to disk, console, or error output. This holds identically for EVERY provider ŌĆö a DeepSeek/Moonshot/MiniMax key gets the same handling as an Anthropic key.
3. **No path traversal.** `caseId` and slugs from the client are resolved ONLY via lookup in the parsed manifest / canonical listing ŌĆö never by joining user input into a filesystem path.
4. **`search_kb` results stay server-side.** The chat response exposes only `kbLookups` (a count). Queries and matched slugs would leak the diagnosis.
5. **Dry-run mode** (`POST /api/examiner?dryRun=1`, returns assembled system blocks without calling Anthropic) must be gated to `process.env.NODE_ENV === 'development'` ŌĆö return 404 otherwise.
6. **The server-held demo key never reaches the client** and is only ever used server-side behind a valid signed demo session ŌĆö see the "Demo access" section below for the gate's own invariants.
7. **The dev CLI bridge is dev-only** (`lib/devCli.ts`, added 2026-07-08): with `NODE_ENV === 'development'` AND `DEV_CLAUDE_CLI=1`, a keyless `/api/examiner` request is served by shelling out to the local `claude -p` CLI (headless Claude Code ŌĆö burns the developer's claude.ai subscription quota instead of API credits, for local tuning). Both conditions must hold; production never takes this path. A BYOK key always takes precedence (keeps the real API path testable). `GET /api/dev-status` (the client's probe for this mode) must 404 outside development. Marking in bridge mode uses plain-JSON output validated by the same shared `lib/marksheet.ts` rules as the tool-use path; final validation of examiner behaviour must still run through the real API path.
8. **Clinical-image reveal is findings-gated** (`lib/images.ts` + `content/case_images.json`, added 2026-07-08): photos are revealed to the candidate ONLY in-chat, by the examiner appending an opaque `{{IMG:im0X}}` marker when it discloses a finding for a region the candidate examined ŌĆö the same discipline as text findings. Never on the stem, never before the region is examined. The caseŌåöimage map is server-only (`getCaseImages`, keyed by caseCode); only a `RevealedImage` (url + spoiler-safe sign-level caption) reaches the client, and only inside an examiner chat reply. Marker ids carry no diagnosis; unknown/duplicate ids are dropped; the client `PublicCase` payload stays `meta`+`stem`. **INTERIM sourcing:** images are extracts from third-party study PDFs ŌĆö gitignored (`/public/case-images/`), noindexed, and to be swapped for open-licensed images before wide release. Provenance + swap worklist: `../_index/case_images.json`.

## Multi-provider rules (binding; added 2026-07-08)

- **The client never controls where a request goes.** The provider is derived server-side from the allowlisted model id; the baseURL comes from the fixed `PROVIDER_CONFIG` map in `lib/providers.ts`. Any request can only ever reach one of the five hosts in that map (gateway + anthropic + deepseek + moonshot + minimax). Adding a provider = a registry entry + a `MODELS` entry, never a client-supplied URL.
- **One BYOK key slot per provider on the client** (`paces.apiKey` stays the Anthropic slot for back-compat; others are `paces.apiKey.<provider>`), all localStorage-only. The examiner call sends the SELECTED model's provider key in the same `API_KEY_HEADER`. Keys must never be folded into the encounter autosave blob (unchanged rule, now plural).
- **Invited access is per-provider:** each provider has its own optional server-held key env var (`DEMO_GATEWAY_API_KEY` ŌĆö recommended, covers both offered models; plus `DEMO_ANTHROPIC_API_KEY`, `DEMO_DEEPSEEK_API_KEY`, `DEMO_MOONSHOT_API_KEY`, `DEMO_MINIMAX_API_KEY` for the direct providers). Demo mode is enabled when Ōēź1 is set; `/api/demo/status` reports the covered providers (post-auth only); a valid session selecting an uncovered provider's model gets a 403 with guidance (an anonymous request still gets the plain 401 ŌĆö no config leakage pre-auth).
- **Capability flags are doc-verified server truth** (`lib/providers.ts`, verified 2026-07-08 against official compat docs): forced `tool_choice` works on Anthropic + DeepSeek only ŌĆö Moonshot/MiniMax mark via the strict-JSON instruction + shared `lib/marksheet.ts` validation (the dev-CLI-bridge pattern; totals recomputed server-side either way). Non-Anthropic providers get `system` flattened to one string (array form undocumented on DeepSeek/Moonshot; their prompt caching is automatic, `cache_control` is ignored there). Moonshot sends `thinking:{type:"disabled"}` (K2.6 defaults thinking ON and bills it as output) and authenticates with `Authorization: Bearer` via the SDK's `authToken` (with `apiKey` pinned null so the env fallback can't fire).
- **Reveal-reminder guardrail:** providers flagged `revealReminder` (all non-Anthropic ŌĆö weaker instruction-following) get a short spoiler-free discipline reminder appended to the LAST user turn of each chat call, transiently and server-side only. The client transcript is never mutated; the reminder must never name case content.
- **Error mapping is provider-labelled but otherwise unchanged:** fixed generic strings parameterised only by the provider display label; upstream error bodies (some compat endpoints emit OpenAI-shaped errors) are never forwarded. A BYOK upstream 401 ŌåÆ 401 "Invalid <Provider> API key"; a demo-key upstream 401 ŌåÆ 502 "invited-access key rejected" (owner's problem, not the user's).
- **The dev CLI bridge covers Anthropic models only** (`claude -p` can't run other providers): a keyless dev request for a non-Anthropic model falls through to the demo/401 path, never the CLI.
- **The gateway is the managed front door (Phase 0):** the `gateway` provider (Vercel AI Gateway) is the only provider surfaced in `MODELS` ŌĆö one `DEMO_GATEWAY_API_KEY` (or a BYOK gateway key) covers both offered models. It obeys every rule here like any other provider (baseURL fixed server-side, key never stored, per-provider demo coverage, capability flags). Its flags: forced tool_choice ON (both underlying models honour it), `system` kept as cache_control blocks (Sonnet caching), reveal-reminder ON (guards DeepSeek).
- **Model ids are pinned, current, and exclusive:** `deepseek-v4-flash` / `deepseek-v4-pro` (the `deepseek-chat`/`deepseek-reasoner` aliases are hard-deprecated 2026-07-24 and must not be used; NB DeepSeek silently maps unknown ids to v4-flash ŌĆö the allowlist is the only typo guard), `kimi-k2.6`, `MiniMax-M3`.

Outputs:
- `content/cases/<sitting>__<filename>` ŌĆö verbatim copy of each case file, flattened name, e.g. `content/cases/2024-03_CGH_Cx__4_Station3_Cardiovascular.md`.
- `content/canonical/<slug>.md` ŌĆö verbatim copies.
- `content/rubric.md` ŌĆö verbatim copy.
- `content/kb_lookup.json` ŌĆö `KbLookup`: from `topic_lookup`, keep only refs with `corpus === "canonical"`, map lowercased term ŌåÆ array of canonical slugs (basename of `file` without `.md`). Drop terms with no canonical ref.
- `content/manifest.json` ŌĆö `Manifest` (see `lib/types.ts`), cases sorted by sitting then encounterNo.

Per-case parsing rules:
- `encounterNo`, `station`, `specialty` from the filename (`4_Station3_Cardiovascular.md` ŌåÆ 4, 3, "Cardiovascular").
- `encounterType`: specialty "Communication" ŌåÆ `communication`; "Consultation" ŌåÆ `consultation`; else `examination`.
- `skills`: parse the line-2 blockquote `skills: A┬ĘB┬ĘD┬ĘE┬ĘG` (middot-separated). Fallback by type: examination ŌåÆ A,B,D,E,G; communication ŌåÆ C,E,F,G; consultation ŌåÆ AŌĆōG. Report any file where parsed skills differ from the fallback expectation (informational, not fatal).
- `timing` by encounterType: examination ŌåÆ "6 min exam + 4 min Q&A"; communication ŌåÆ "5 min reading + 10 min"; consultation ŌåÆ "5 min reading + 15 min + 5 min Q&A".
- `sittingLabel`: from sitting dir name ŌĆö `2024-03_CGH_Cx` ŌåÆ "CGH ┬Ę Mar 2024". Format: `<HOSPITAL> ┬Ę <Mon> <YYYY>`; the dir-name suffix (`_Cx` / `_CycleN` / `_AM` / `_PM`) is validated but DROPPED (a format artifact ŌĆö same-month sittings share a label and merge in the picker). Library cases use the collection's `LIBRARY_LABELS` source name instead.
- `displayTitle`: the filename `specialty` verbatim ŌĆö the exam system ("Cardiovascular" / "Respiratory" / "Neurology" / "Abdominal") for examination cases, else the classification ("Communication" / "Consultation"). **Station numbers appear NOWHERE in the UI** (per Arthur 2026-07-06 ŌĆö they are format artifacts; the `station` field stays server/meta-only). From filename metadata ONLY (invariant 1).
- Stem extraction (validation only ŌĆö the stem is served at runtime by Agent B's parser, but Agent A must verify every file HAS one): section from the `## Candidate stem` heading (prefix match) to the next `## ` heading, then (identically to the runtime parser) spoiler-note blocks stripped AND presentation-normalized for the near-plain-text stem card: blockquote markers ("> ") unwrapped, horizontal-rule lines ("---") dropped. Fail the build loudly listing any file without a non-empty stem.
- `caseCode`: looked up from `../_index/case_ids.json` by app case id ŌĆö the user-facing case ID (client-visible by design; opaque, no content/provenance meaning). Build FAILS on a missing code, and on duplicate codes across served cases.
- `canonicalSlugs`: match the case's H1 title (+ first blockquote line) against every canonical note's `condition` + `aliases`: case-insensitive, word-boundary-ish substring (alias appears in title text); consider only aliases/conditions Ōēź 6 chars to avoid junk matches; score by matched-string length; keep top 3 slugs max. Empty is fine (most communication cases). Report the match-rate per encounterType.

Report back: case count (must be 503 = 287 carousel + 216 library, with carousel 287 + placeholders 9 === 296), canonical count (must be Ōēź 156), caseCode coverage (every served case; all unique), kb term count, stem-extraction failures (must be 0), canonical match-rate by type, and 3 sample manifest entries.

## Agent B ŌĆö backend

### `lib/content.ts` ŌĆö server-only loaders (fs reads from `path.join(process.cwd(), 'content')`), each cached in a module-level variable:
- `getManifest(): Manifest`, `getCaseMeta(id): CaseMeta | undefined`
- `getCaseFull(id): string` ŌĆö full case markdown (server-side only)
- `getCaseStem(id): string` ŌĆö the `## Candidate stem` section only (heading prefix-match to next `## `), stripped of the heading line itself
- `getCanonicalNote(slug): string | undefined` ŌĆö only slugs present in the canonical dir listing (build a Set at startup; invariant 3)
- `getRubric(): string`

### `lib/kb.ts` ŌĆö the `search_kb` implementation:
- Load `content/kb_lookup.json` once. `searchKb(query: string): { slugs: string[], text: string }`.
- Normalize query (lowercase, trim, collapse spaces). Match: exact term hit first; else terms that contain the query or vice-versa (substring, both directions); score by overlap length; collect top canonical slugs (max 2 distinct notes).
- Return the concatenated note contents (each prefixed `--- canonical/<slug>.md ---`), total capped at ~20,000 chars; empty-result message "No canonical note matched. Answer from the case brief and general medical knowledge, flagging uncertainty." if nothing matches.

### `lib/prompt.ts` ŌĆö prompt assembly:
`buildSystem(meta: CaseMeta): TextBlockParam[]` returning exactly two system blocks, each with `cache_control: { type: 'ephemeral' }`:

**Block 1 (static across all cases):** the examiner persona below, with the rubric (from `getRubric()`) appended verbatim under the heading `# MARKING RUBRIC (apply exactly)`.

The persona text (verbatim, this is the product's core ŌĆö do not paraphrase):

```
You are an MRCP PACES examiner running one practice encounter with a candidate, over text chat. For communication and consultation encounters you ALSO play the simulated patient, relative, or colleague. The human is always the candidate ŌĆö never speak for them. Realistic, professional, exam-standard throughout. British English, standard PACES conventions.

THE ENCOUNTER BRIEF in the next system block contains: the encounter type and skills marked, the full case file (stem, expected findings keyed to examination manoeuvres, model presentation, viva questions with model answers, hidden answer key), and reference grounding notes. The candidate has already read the candidate stem ŌĆö do not re-read it to them.

YOUR ROLE BY ENCOUNTER TYPE:
- EXAMINATION encounter (Respiratory / Cardiovascular / Neurology / Abdominal; skills A┬ĘB┬ĘD┬ĘE┬ĘG): you are the examiner invigilating a 6-minute physical examination. The candidate tells you what they examine or look for, step by step. For each manoeuvre they actually perform, reveal EXACTLY the findings the case file keys to that manoeuvre ŌĆö nothing more, nothing sooner, faithful to the file's wording of the clinical signs. If they perform a manoeuvre the file doesn't cover, give a realistic noncontributory/normal finding without inventing diagnostic facts. Do not volunteer findings for steps they skipped; missed signs are marked, not rescued (do not prompt or hint, beyond a single neutral "is there anything else you wish to examine?" when they say they are done). When they finish (or the exchange clearly reaches examination's end), ask them to present their findings and diagnosis. Then run the viva from the case file's questions, adapting to what they missed. Keep each of your turns short and examiner-like.
- COMMUNICATION encounter (skills C┬ĘE┬ĘF┬ĘG): play the character in the case file's surrogate brief ŌĆö their knowledge, emotions, and behavioural rules (including how they respond to a good vs poor approach). Stay strictly in character; reveal facts only as the brief allows and only when the candidate's approach earns them. Press with the character's scripted questions and concerns at natural moments. Do NOT slip into examiner voice until the candidate says they are finished (or clearly closes the conversation) ŌĆö then debrief.
- CONSULTATION encounter (all 7 skills AŌĆōG): play the patient from the brief for the history; answer only from the brief's facts ("I'm not sure, doctor" for anything not covered). When the candidate examines, reveal findings per the examination rules above. After their presentation, switch to examiner Q&A from the case file.

GOLDEN RULES:
1. NEVER reveal, hint at, or confirm the diagnosis, the answer key, the model presentation, or model viva answers until the candidate has committed to a presentation and diagnosis (or explicitly ends the encounter). This includes indirect leaks: over-specific vocabulary, leading questions, or confirming a guess mid-encounter.
2. Use only the facts in the encounter brief. Never invent new clinical facts, results, or history. Patients don't know medical terminology they wouldn't plausibly know.
3. Findings are earned: reveal strictly manoeuvre-by-manoeuvre / question-by-question.
4. Once the candidate has presented and the viva is done (or they give up), switch to TUTOR DEBRIEF: state the diagnosis and key discriminators, compare their presentation with the model presentation, list what they found, missed, or invented, and give focused teaching from the viva material and grounding notes.
5. In viva and debrief, ground factual specifics (management, investigations, guidelines) in the encounter brief and grounding notes; use the search_kb tool for tangents they don't cover rather than answering from memory. Flag any residual uncertainty honestly.
6. When asked to mark (the marksheet request), grade per the rubric below: ONLY the skills marked for this encounter; apply the "skip the courtesies" rules exactly (never score or nag about hand hygiene, introductions, consent, draping); weight B (signs) and D (diagnosis) heavily for examination encounters; every Borderline or Unsatisfactory needs a one-line justification tied to the descriptors; name the single change that would most improve the weakest skill.
7. If the candidate types "[BEGIN ENCOUNTER]", open the encounter in role: examinations ŌåÆ invite them to begin examining; communication/consultation ŌåÆ open with the scene and, where the brief scripts one, the character's opening line.
```

**Block 2 (per case):** header lines `ENCOUNTER: <displayTitle> ┬Ę <sittingLabel> ┬Ę type=<encounterType> ┬Ę skills=<skills> ┬Ę timing=<timing>`, then `# CASE FILE (hidden from candidate ŌĆö the candidate has seen ONLY the "Candidate stem" section)` + full case markdown, then for each canonical slug `# REFERENCE GROUNDING: <slug> (for viva/debrief accuracy; never read out verbatim)` + note text. Cap block 2 at ~150,000 chars (truncate grounding notes first, never the case file).

### API routes (all `export const runtime = 'nodejs'`):
- `GET /api/manifest` ŌåÆ `PublicManifest` (200) ŌĆö the full `Manifest` (with `canonicalSlugs`) stays server-side for prompt assembly (invariant 1).
- `GET /api/case/[id]` ŌåÆ `PublicCase` or 404 `ApiError`.
- `POST /api/examiner` ŌåÆ body `ExaminerRequest`, key from `API_KEY_HEADER` (401 `ApiError` if missing/blank; 400 on unknown caseId, model not in `MODEL_ALLOWLIST`, empty messages, messages > 400, or any message > 30,000 chars).
  - `action: 'chat'`: `messages.create` with the two system blocks, transcript as-is, `max_tokens: 1500`, tools: [`search_kb` ŌĆö description: "Look up a condition, sign, or topic in the PACES reference library. Returns the canonical grounding note. Use for viva/debrief tangents not covered by the encounter brief."; input schema `{ query: string }`]. Tool-use loop: while `stop_reason === 'tool_use'` (max 3 iterations) run `searchKb`, append tool_result, re-call. Reply = concatenated text blocks. Respond `ExaminerChatResponse` with real `usage` (sum across iterations; map `cache_read_input_tokens`/`cache_creation_input_tokens`).
  - `action: 'mark'`: same system blocks; append user turn "Please complete the marksheet for this encounter now, based strictly on the transcript so far."; single tool `submit_marksheet` (input schema mirroring `MarkSheet`; skills enum AŌĆōG, grade enum 0/1/2), `tool_choice: { type: 'tool', name: 'submit_marksheet' }`, `max_tokens: 2500`. Validate: skills ŌŖå the case's marked skills, recompute `total`/`maxTotal` server-side. Respond `ExaminerMarkResponse`.
  - Anthropic errors: map 401 ŌåÆ 401 "Invalid API key", 429 ŌåÆ 429 "Rate limited by Anthropic", else 502 with the SDK's error message ONLY if it cannot contain case content (use a generic message otherwise). Never include prompts in errors.

## Demo access (whitelisted magic-link sign-in)

Purpose: let a small set of invited users (whitelisted by email) practise on **server-held, spend-capped** per-provider keys with zero BYOK friction, while keeping those keys unusable by anyone else. Entirely optional: with the env unset the feature is invisible and the app stays BYOK-only. Implementation: `lib/demo.ts` (server-only), `app/api/demo/{request,verify,status}/route.ts`, `components/DemoAccess.tsx`; `nodemailer` is the feature's only dependency. In the UI the sidebar section is labelled **"Invited access"** ("demo" is the owner's framing, not the invitee's) and it renders ABOVE Settings; the `?demo=active|invalid` flag appended by verify's redirect is handled by `page.tsx` as a main-pane banner (visible on mobile, where the sidebar is off-canvas), and a keyless 401 from `/api/examiner` is surfaced as sign-in-expired guidance plus a demo-status re-fetch, never the raw "Missing API key".

**Env** (documented in `.env.example` / README): `DEMO_WHITELIST` (comma-separated emails, matched case-insensitive + trimmed), per-provider server-held keys `DEMO_GATEWAY_API_KEY` (recommended ŌĆö covers both offered models) / `DEMO_ANTHROPIC_API_KEY` / `DEMO_DEEPSEEK_API_KEY` / `DEMO_MOONSHOT_API_KEY` / `DEMO_MINIMAX_API_KEY` (all unset = demo mode off; a session can only use models whose provider has a key set ŌĆö see "Multi-provider rules"), `AUTH_SECRET` (HMAC secret, required), `SMTP_HOST`/`SMTP_PORT`/`SMTP_USER`/`SMTP_PASS` (unset ŌćÆ **in development** the magic link is logged to the server console instead of emailed; **in production** a missing or partial SMTP config logs a loud error and the link is NOT sent and NOT logged ŌĆö a live sign-in credential must never land in deployment logs; never a hard failure either way), `APP_BASE_URL` (magic-link base; **required in production** ŌĆö emailed links are never built from the request-derived origin there, because Host / x-forwarded-host is sender-controlled and a poisoned host would deliver a whitelisted victim's real link token to an attacker domain; if unset in production, link generation fails closed with a server-side error while the generic 200 still goes out. Development falls back to the request origin so the local flow needs no config. The request-derived origin remains fine for verify's same-origin redirects).

**Tokens** are stateless HMAC signatures (no DB): `base64url(JSON{email, purpose, exp}) + '.' + base64url(HMAC-SHA256(AUTH_SECRET, payload))`. Two disjoint purposes, checked on verification: `link` (emailed, 15 min) and `session` (httpOnly cookie `demo_session`, 30 days) ŌĆö so a captured session cookie can never be replayed through `/api/demo/verify` to mint fresh sessions.

**Flow**
1. `POST /api/demo/request` `{email}` ŌåÆ **always** the same generic 200 (`DEMO_REQUEST_MESSAGE` in `lib/types.ts`), whitelisted or not, configured or not. If (and only if) demo mode is fully configured, the email is whitelisted, AND a trusted link base is available (`APP_BASE_URL`, or the request origin in dev ŌĆö see env above), a `link` token is signed and the magic link `{base}/api/demo/verify?token=ŌĆ”` is emailed ŌĆö inside `after()`, so response timing can't leak the branch. Best-effort in-memory rate limit: 5/hour per email and 30/hour per IP (looser per-IP because hospital NATs put many invitees behind one address, and `x-forwarded-for` is only meaningful behind a trusted proxy such as Vercel), applied *before* the whitelist check (a 429 carries no membership signal). Both buckets are checked before either consumes a slot, so an IP-rejected burst can't drain an email's allowance. Known, accepted tradeoff: a third party who knows an invited address can fill that email's bucket (temporary sign-in lockout) ŌĆö scoping the count to whitelisted emails would turn the 429 threshold into a membership oracle. Acceptable as best-effort in serverless because the tokens are unguessable anyway.
2. `GET /api/demo/verify?token=` ŌåÆ constant-time signature check (both comparands SHA-256-hashed first, then `timingSafeEqual`), expiry check, purpose=`link` check, and the email must **still** be whitelisted at verify time. Success: set `demo_session` (httpOnly, Secure in production, SameSite=Lax, 30 d) containing a fresh `session` token and redirect to `/?demo=active`. **Every** failure redirects to `/?demo=invalid` ŌĆö no detail about which check failed.
3. `GET /api/demo/status` ŌåÆ `{active: boolean, email?: masked "c***@domain", providers?: ProviderId[]}` from the cookie, for UI state only. `active` requires the full working path: valid signed cookie + still whitelisted + at least one `DEMO_*_API_KEY` set; `providers` lists the covered providers (post-auth only).
4. `POST /api/examiner`: a client-supplied BYOK key (`API_KEY_HEADER`) takes precedence exactly as before; else a **valid** `demo_session` cookie (signed, unexpired, still-whitelisted) + the SELECTED MODEL'S provider `DEMO_*_API_KEY` set ŌćÆ that server key is used; a valid session whose providers don't cover the model gets 403 with guidance; else the existing 401 "Missing API key". An upstream 401 on a demo key maps to a 502 "invited-access key rejected by <Provider>" (owner's problem), not 401 "Invalid API key" (which would send a keyless user hunting for a key).

**Invariants (violations are critical bugs, same standing as the list at the top)**
- The whitelist is enforced at request time AND verify time AND every use of the session (status + examiner) ŌĆö removing an email from `DEMO_WHITELIST` revokes outstanding links and cookies immediately.
- All demo endpoints answer generically: no response (status code, body, redirect target, or timing) may reveal whether an email is whitelisted or why a token failed.
- No `DEMO_*_API_KEY` is ever sent to, or readable by, the client in any form ŌĆö each goes from `process.env` straight into the Anthropic SDK constructor server-side, under invariant 2's no-store/no-log rules, and only behind a valid signed cookie.
- BYOK precedence is absolute: a request carrying `API_KEY_HEADER` never touches the server key.
- The demo gate changes nothing else: answer-key hiding (invariant 1), `PublicCaseMeta` projection, model allowlist, and all other existing invariants hold unchanged with or without a demo session.
