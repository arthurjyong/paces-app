# PACES AI-Examiner MVP ŌĆö build spec (binding contract)

One screen (chat + case picker + BYOK key box) + one examiner backend function + the existing 503-case corpus (287 carousel encounters + 216 standalone library cases). A candidate picks a case, the AI plays PACES examiner/patient grounded on the case file + canonical notes, reveals findings manoeuvre-by-manoeuvre, runs the viva, and marks against the official rubric with structured output. Stack: Next.js 16 (App Router, TypeScript, Tailwind v4) + `@anthropic-ai/sdk`. Deployable to Vercel; runs locally with `npm run dev`.

**Multi-provider adapter (added 2026-07-08):** the same `@anthropic-ai/sdk` code path serves several providers whose endpoints speak the Anthropic Messages API ŌĆö Anthropic, DeepSeek, Moonshot (Kimi), MiniMax ŌĆö giving the app a budget tier ~2ŌĆō50├Ś cheaper per case depending on model (DeepSeek Ōēł 30ŌĆō50├Ś, MiniMax Ōēł 5ŌĆō8├Ś, Kimi Ōēł 2ŌĆō3├Ś). See "Multi-provider rules" below the invariants.

**Managed gateway path (Phase 0, added 2026-07-09):** the direction is a MANAGED tier where the operator tops up **one** place that fans out across models. That place is **Vercel AI Gateway** ŌĆö added as the `gateway` provider (baseURL `https://ai-gateway.vercel.sh`, Anthropic-Messages surface, x-api-key auth, key env `DEMO_GATEWAY_API_KEY`), fronting both offered models at **zero token markup**: `anthropic/claude-sonnet-4.6` (premium) and `deepseek/deepseek-v4-pro` (budget). `MODELS` is deliberately trimmed to just these two (Arthur, 2026-07-09: "only Sonnet 4.6 vs DeepSeek Pro at the beginning"). *(Re-expanded to 8 entries in Phase 1 the same day ŌĆö the 2 gateway models + 3 Anthropic BYOK-direct + 3 OpenRouter BYOK; see "Multi-provider rules".)* The four direct providers stay wired in `PROVIDER_CONFIG` for BYOK-direct and as the off-Vercel escape hatch, but are not surfaced in `MODELS`. **DEPLOYED & LIVE in production 2026-07-09** (https://paces-app.vercel.app ŌĆö prod routes both offered models through the gateway via `DEMO_GATEWAY_API_KEY`; old direct model slugs now 400). ## Nomenclature (binding ŌĆö the three case axes; locked with Arthur 2026-07-06)

Every case is described by three INDEPENDENT axes. Never mix them in labels, filters, or copy:

1. **Classification** ŌĆö what kind of encounter it is: `consultation` | `communication` | `examination` (the `encounterType` field). This is the picker's default grouping and its filter chips. (Consult = old "Station 5" = brief clinical consultation ŌĆö one skill; station numbers are format artifacts and never appear on consult labels.)
2. **Theme** ŌĆö the clinical topic (`theme` field: cardiology, ophthalmology, nephrology, endocrine, ŌĆ”; curated in `../_index/case_themes.json`). An attribute, not a station. The four exam systems are both a station format and a theme. UI: tick-box multi-select with Select all.
3. **Source** ŌĆö for carousel cases: the real sitting ("NUH ┬Ę Mar 2026"). For ALL standalone cases: one pooled type-named bank ŌĆö "Consult bank" / "Communication bank" / "Examination bank" (per Arthur 2026-07-06: collection provenance is deliberately NOT user-visible; it stays internal). Source labels must never smuggle in content/topic ("Eye OSCEs" was wrong ŌĆö the eye-ness is theme=ophthalmology), author names, or initials ŌĆö and collection dir names flow into served `id`/`sitting`/`file` fields, so the DIR names must be neutral too.

`specialty` (filename-derived: Cardiovascular/ŌĆ”/Communication/Consultation) is a legacy field that mixes axes 1 and 2 ŌĆö keep it server/meta-only for exam-station grouping; don't build new UI on it.

## Canonical candidate-stem format (binding ŌĆö locked with Arthur 2026-07-07; all 503 source stems rewritten to it)

The `## Candidate stem` section of every case file is **"official exam prose"**: 2ŌĆō4 short plain-prose paragraphs in the style of the real MRCP PACES written instruction. **Near-plain text only** ŌĆö no bold, bullets, blockquotes, `---`, headings, tables. No tutor/meta commentary, no encounter-timing sentences (the card header shows authoritative timing), no station numbers, no diagnosis reveals (given diagnoses in communication scenarios are fine ŌĆö they're scenario information).

- **Consultation:** "You are the [role] in [setting]." ŌåÆ patient + quoted referral ŌåÆ "On arrival: [vitals prose]" (if given) ŌåÆ "Your task is to assess the patient's problems and address any questions or concerns raised by the patient. You will then discuss the case with the examiners."
- **Communication:** "You are the [role] in [setting]." ŌåÆ "You are about to speak to [person]." + door-given background ŌåÆ available information (if given) ŌåÆ "Your task is to [task]. You will then discuss the case with the examiners."
- **Examination (brief):** "This [age]-year-old [man/woman] [one-line context]." ŌåÆ "Please examine [system] and then present your findings, with your diagnosis and differential. Do not take a history."

Any future enrichment/generation MUST emit stems in this format ŌĆö the runtime spoiler/timing/artifact strips in `getCaseStem()` are safety nets, not the convention (as of the 2026-07-07 rewrite they strip 0 blocks corpus-wide).
**Shared types:** `lib/types.ts` (already written ŌĆö code against it, never redefine its types).

## Security invariants (any violation is a critical bug)

1. **Hidden content never reaches the client.** The client may only ever receive: the manifest (`PublicCaseMeta` fields ŌĆö `CaseMeta` minus `canonicalSlugs` and `file`; `canonicalSlugs` is matched from the case H1 title so the slugs name the diagnosis, per invariant 4's rationale), the `stem` of a case, the examiner model's chat replies, and the marksheet. The case H1 title, expected findings, model presentation, viva answers, answer key, surrogate brief, canonical notes, and assembled prompts must NEVER appear in any client-visible payload, including error messages. `displayTitle` is built from filename metadata only (the H1 title names the diagnosis ŌĆö it is a spoiler).
2. **The user's API key is never stored or logged server-side.** It arrives per-request in the `API_KEY_HEADER` header, is passed straight to the Anthropic SDK constructor (as `apiKey` or, for Bearer-auth providers, `authToken`), and is never written to disk, console, or error output. This holds identically for EVERY provider ŌĆö a DeepSeek/Moonshot/MiniMax key gets the same handling as an Anthropic key.
3. **No path traversal.** `caseId` and slugs from the client are resolved ONLY via lookup in the parsed manifest / canonical listing ŌĆö never by joining user input into a filesystem path.
4. **`search_kb` results stay server-side.** The chat response exposes only `kbLookups` (a count). Queries and matched slugs would leak the diagnosis.
5. **Dry-run mode** (`POST /api/examiner?dryRun=1`, returns assembled system blocks without calling Anthropic) must be gated to `process.env.NODE_ENV === 'development'` ŌĆö return 404 otherwise.
6. **The server-held gateway key never reaches the client** and is only ever used server-side behind a valid signed managed session ŌĆö see the "Managed access" section below for the gate's own invariants. *(Until 2026-07-09 this read "demo key ŌĆ” demo session"; the demo gate is superseded ŌĆö see the tombstone below.)*
7. **The dev CLI bridge is dev-only** (`lib/devCli.ts`, added 2026-07-08): with `NODE_ENV === 'development'` AND `DEV_CLAUDE_CLI=1`, a keyless `/api/examiner` request is served by shelling out to the local `claude -p` CLI (headless Claude Code ŌĆö burns the developer's claude.ai subscription quota instead of API credits, for local tuning). Both conditions must hold; production never takes this path. A BYOK key always takes precedence (keeps the real API path testable). `GET /api/dev-status` (the client's probe for this mode) must 404 outside development. Marking in bridge mode uses plain-JSON output validated by the same shared `lib/marksheet.ts` rules as the tool-use path; final validation of examiner behaviour must still run through the real API path.
8. **Clinical-image reveal is findings-gated** (`lib/images.ts` + `content/case_images.json`, added 2026-07-08): photos are revealed to the candidate ONLY in-chat, by the examiner appending an opaque `{{IMG:im0X}}` marker when it discloses a finding for a region the candidate examined ŌĆö the same discipline as text findings. Never on the stem, never before the region is examined. The caseŌåöimage map is server-only (`getCaseImages`, keyed by caseCode); only a `RevealedImage` (url + spoiler-safe sign-level caption) reaches the client, and only inside an examiner chat reply. Marker ids carry no diagnosis; unknown/duplicate ids are dropped; the client `PublicCase` payload stays `meta`+`stem`. **INTERIM sourcing:** images are extracts from third-party study PDFs ŌĆö gitignored (`/public/case-images/`), noindexed, and to be swapped for open-licensed images before wide release. Provenance + swap worklist: `../_index/case_images.json`.
9. **The server gateway key is only ever used behind a valid managed session whose LIVE database tier covers the requested model AND whose spend reservation succeeded** (added 2026-07-09 ŌĆö THE Phase-1 abuse fix). There is no anonymous server-key path of any kind: a request with no valid `paces_session` cookie, a session whose `email_overrides`/`allowed_domains` rows have been removed, a model outside `TIER_MODELS[tier]`, or a failed `reserveSpend` all stop BEFORE the SDK client is constructed. A BYOK request never touches the server key at all (precedence is absolute).
10. **Metering is server-truth** (added 2026-07-09). `usage_events` is append-only; `user_balances` and `global_spend` move ONLY through `reserveSpend`/`settleSpend` (atomic conditional UPDATEs); costs come from the server-side price table (`lib/pricing.ts`) applied to the provider's real `usage`. No client input ŌĆö model string, transcript, headers, retries ŌĆö can lower a charge, dodge a reservation, or touch another user's meter.

## Multi-provider rules (binding; added 2026-07-08)

- **The client never controls where a request goes.** The provider is derived server-side from the allowlisted model id; the baseURL comes from the fixed `PROVIDER_CONFIG` map in `lib/providers.ts`. Any request can only ever reach one of the six hosts in that map (gateway + anthropic + openrouter + deepseek + moonshot + minimax). Adding a provider = a registry entry + a `MODELS` entry, never a client-supplied URL.
- **One BYOK key slot per provider on the client** (`paces.apiKey` stays the Anthropic slot for back-compat; others are `paces.apiKey.<provider>`), all localStorage-only. The examiner call sends the SELECTED model's provider key in the same `API_KEY_HEADER`. Keys must never be folded into the encounter autosave blob (unchanged rule, now plural).
- **Server-held keys are managed-door-only, and the managed door is gateway-only** *(supersedes the per-provider invited-access rule, 2026-07-09)*: the only server-held key any live code path spends is `DEMO_GATEWAY_API_KEY` (env name kept from Phase 0), and only behind a valid managed session + live tier + successful reservation (see "Managed access" and invariant 9). The direct-provider vars (`DEMO_ANTHROPIC_API_KEY`, `DEMO_OPENROUTER_API_KEY`, `DEMO_DEEPSEEK_API_KEY`, `DEMO_MOONSHOT_API_KEY`, `DEMO_MINIMAX_API_KEY`) stay named in `PROVIDER_CONFIG` as the BYOK-direct / off-Vercel escape hatch but are unused by the managed door. The `/api/demo/*` surface is retired; a valid session selecting a model its tier doesn't cover gets a 403 with guidance (an anonymous request still gets the plain 401 ŌĆö no config leakage pre-auth).
- **Capability flags are doc-verified server truth** (`lib/providers.ts`, verified 2026-07-08 against official compat docs): forced `tool_choice` works on Anthropic + DeepSeek only ŌĆö Moonshot/MiniMax mark via the strict-JSON instruction + shared `lib/marksheet.ts` validation (the dev-CLI-bridge pattern; totals recomputed server-side either way). Non-Anthropic providers get `system` flattened to one string (array form undocumented on DeepSeek/Moonshot; their prompt caching is automatic, `cache_control` is ignored there). Moonshot sends `thinking:{type:"disabled"}` (K2.6 defaults thinking ON and bills it as output) and authenticates with `Authorization: Bearer` via the SDK's `authToken` (with `apiKey` pinned null so the env fallback can't fire).
- **Reveal-reminder guardrail:** providers flagged `revealReminder` (all non-Anthropic ŌĆö weaker instruction-following) get a short spoiler-free discipline reminder appended to the LAST user turn of each chat call, transiently and server-side only. The client transcript is never mutated; the reminder must never name case content.
- **Error mapping is provider-labelled but otherwise unchanged:** fixed generic strings parameterised only by the provider display label; upstream error bodies (some compat endpoints emit OpenAI-shaped errors) are never forwarded. A BYOK upstream 401 ŌåÆ 401 "Invalid <Provider> API key"; a server-key upstream 401 ŌåÆ 502 with a fixed managed-key-rejected string (the operator's problem, not the user's ŌĆö a 401 would send a keyless user hunting for a key).
- **The dev CLI bridge covers Anthropic models only** (`claude -p` can't run other providers): a keyless dev request for a non-Anthropic model falls through to the managed/401 path, never the CLI.
- **The gateway is the managed front door** (Phase 0 = the money layer; Phase 1 gates it behind login + tier + meter ŌĆö see "Managed access"): the `gateway` provider (Vercel AI Gateway) is the ONLY provider the managed door routes through ŌĆö one `DEMO_GATEWAY_API_KEY` (or a BYOK gateway key) covers both gateway models. Since Phase 1, `MODELS` also surfaces `anthropic` (BYOK-direct Claude lineup: Sonnet 4.6 / Opus 4.8 / Haiku 4.5) and `openrouter` (curated shortlist) ŌĆö 8 entries total; the managed door never touches those. The gateway obeys every rule here like any other provider (baseURL fixed server-side, key never stored, capability flags). Its flags: forced tool_choice ON (both underlying models honour it), `system` kept as cache_control blocks (Sonnet caching), reveal-reminder ON (guards DeepSeek).
- **OpenRouter is the promoted BYOK aggregator (Phase 1, added 2026-07-09):** baseURL `https://openrouter.ai/api` ŌĆö the SDK appends `/v1/messages` itself (the docs pin the baseURL WITHOUT `/v1`; adding it would produce `/v1/v1/messages`); Bearer auth via the SDK's `authToken` option (`apiKey` pinned null, same pattern as Moonshot). Forced `tool_choice` and `cache_control` system blocks verified supported 2026-07-09 (the endpoint translates both for ANY underlying model, GPT included; Anthropic-model caching passes through) ŌĆö so marking uses the forced tool and the two system blocks stay intact. Offered as a **curated 3-model shortlist** (Claude Sonnet 4.6 / GPT-5.5 / DeepSeek V4 Pro ŌĆö one user key reaches all three), deliberately NOT the full ~300-model catalog. Reveal-reminder ON (the shortlist includes non-Claude models).
- **Registry ids vs wire ids:** the client selects, stores, and sends `MODELS` REGISTRY ids, unique across all providers; the server maps to the upstream slug via `modelWireId()` at call time. Where the same upstream slug exists under two providers (gateway and OpenRouter both speak `anthropic/claude-sonnet-4.6`), the registry id is prefixed (`openrouter/anthropic/claude-sonnet-4.6`, ŌĆ”) and `wireId` carries the real slug ŌĆö a registry id therefore names exactly one provider route, and the allowlist can never be ambiguous about where a request goes.
- **Model ids are pinned, current, and exclusive:** `deepseek-v4-flash` / `deepseek-v4-pro` (the `deepseek-chat`/`deepseek-reasoner` aliases are hard-deprecated 2026-07-24 and must not be used; NB DeepSeek silently maps unknown ids to v4-flash ŌĆö the allowlist is the only typo guard), `kimi-k2.6`, `MiniMax-M3`.

Outputs:

- `content/canonical/<slug>.md` ŌĆö verbatim copies.
- `content/rubric.md` ŌĆö verbatim copy.
- `content/kb_lookup.json` ŌĆö `KbLookup`: from `topic_lookup`, keep only refs with `corpus === "canonical"`, map lowercased term ŌåÆ array of canonical slugs (basename of `file` without `.md`). Drop terms with no canonical ref.

Per-case parsing rules:
- `encounterNo`, `station`, `specialty` from the filename (`4_Station3_Cardiovascular.md` ŌåÆ 4, 3, "Cardiovascular").
- `encounterType`: specialty "Communication" ŌåÆ `communication`; "Consultation" ŌåÆ `consultation`; else `examination`.
- `skills`: parse the line-2 blockquote `skills: A┬ĘB┬ĘD┬ĘE┬ĘG` (middot-separated). Fallback by type: examination ŌåÆ A,B,D,E,G; communication ŌåÆ C,E,F,G; consultation ŌåÆ AŌĆōG. Report any file where parsed skills differ from the fallback expectation (informational, not fatal).
- `timing` by encounterType: examination ŌåÆ "6 min exam + 4 min Q&A"; communication ŌåÆ "5 min reading + 10 min"; consultation ŌåÆ "5 min reading + 15 min + 5 min Q&A".
- `sittingLabel`: from sitting dir name ŌĆö `2024-03_CGH_Cx` ŌåÆ "CGH ┬Ę Mar 2024". Format: `<HOSPITAL> ┬Ę <Mon> <YYYY>`; the dir-name suffix (`_Cx` / `_CycleN` / `_AM` / `_PM`) is validated but DROPPED (a format artifact ŌĆö same-month sittings share a label and merge in the picker). Library cases use the collection's `LIBRARY_LABELS` source name instead.
- `displayTitle`: the filename `specialty` verbatim ŌĆö the exam system ("Cardiovascular" / "Respiratory" / "Neurology" / "Abdominal") for examination cases, else the classification ("Communication" / "Consultation"). **Station numbers appear NOWHERE in the UI** (per Arthur 2026-07-06 ŌĆö they are format artifacts; the `station` field stays server/meta-only). From filename metadata ONLY (invariant 1).
- Stem extraction (validation only ŌĆö the stem is served at runtime by Agent B's parser, but Agent A must verify every file HAS one): section from the `## Candidate stem` heading (prefix match) to the next `## ` heading, then (identically to the runtime parser) spoiler-note blocks stripped AND presentation-normalized for the near-plain-text stem card: blockquote markers ("> ") unwrapped, horizontal-rule lines ("---") dropped. Fail the build loudly listing any file without a non-empty stem.
- `caseCode`: looked up from `../_index/case_ids.json` by app case id ŌĆö the user-facing case ID (client-visible by design; opaque, no content/provenance meaning). Build FAILS on a missing code, and on duplicate codes across served cases.
- `canonicalSlugs`: match the case's H1 title (+ first blockquote line) against every canonical note's `condition` + `aliases`: case-insensitive, word-boundary-ish substring (alias appears in title text); consider only aliases/conditions Ōēź 6 chars to avoid junk matches; score by matched-string length; keep top 3 slugs max. Empty is fine (most communication cases). Report the match-rate per encounterType.

Report back: case count (must be 503 = 287 carousel + 216 library, with carousel 287 + placeholders 9 === 296), canonical count (must be Ōēź 156), caseCode coverage (every served case; all unique), kb term count, stem-extraction failures (must be 0), canonical match-rate by type, and 3 sample manifest entries.

## Agent B ŌĆö backend

### `lib/content.ts` ŌĆö server-only loaders (fs reads from `path.join(process.cwd(), 'content')`), each cached in a module-level variable:
- `getManifest(): Manifest`, `getCaseMeta(id): CaseMeta | undefined`
- `getCaseFull(id): string` ŌĆö full case markdown (server-side only)
- `getCaseStem(id): string` ŌĆö the `## Candidate stem` section only (heading prefix-match to next `## `), stripped of the heading line itself
- `getCanonicalNote(slug): string | undefined` ŌĆö only slugs present in the canonical dir listing (build a Set at startup; invariant 3)
- `getRubric(): string`

### `lib/kb.ts` ŌĆö the `search_kb` implementation:
- Load `content/kb_lookup.json` once. `searchKb(query: string): { slugs: string[], text: string }`.
- Normalize query (lowercase, trim, collapse spaces). Match: exact term hit first; else terms that contain the query or vice-versa (substring, both directions); score by overlap length; collect top canonical slugs (max 2 distinct notes).
- Return the concatenated note contents (each prefixed `--- canonical/<slug>.md ---`), total capped at ~20,000 chars; empty-result message "No canonical note matched. Answer from the case brief and general medical knowledge, flagging uncertainty." if nothing matches.

### `lib/prompt.ts` ŌĆö prompt assembly:
`buildSystem(meta: CaseMeta): TextBlockParam[]` returning exactly two system blocks, each with `cache_control: { type: 'ephemeral' }`:

**Block 1 (static across all cases):** the examiner persona below, with the rubric (from `getRubric()`) appended verbatim under the heading `# MARKING RUBRIC (apply exactly)`.

The persona text (verbatim, this is the product's core ŌĆö do not paraphrase):

```
You are an MRCP PACES examiner running one practice encounter with a candidate, over text chat. For communication and consultation encounters you ALSO play the simulated patient, relative, or colleague. The human is always the candidate ŌĆö never speak for them. Realistic, professional, exam-standard throughout. British English, standard PACES conventions.

THE ENCOUNTER BRIEF in the next system block contains: the encounter type and skills marked, the full case file (stem, expected findings keyed to examination manoeuvres, model presentation, viva questions with model answers, hidden answer key), and reference grounding notes. The candidate has already read the candidate stem ŌĆö do not re-read it to them.

YOUR ROLE BY ENCOUNTER TYPE:
- EXAMINATION encounter (Respiratory / Cardiovascular / Neurology / Abdominal; skills A┬ĘB┬ĘD┬ĘE┬ĘG): you are the examiner invigilating a 6-minute physical examination. The candidate tells you what they examine or look for, step by step. For each manoeuvre they actually perform, reveal EXACTLY the findings the case file keys to that manoeuvre ŌĆö nothing more, nothing sooner, faithful to the file's wording of the clinical signs. If they perform a manoeuvre the file doesn't cover, give a realistic noncontributory/normal finding without inventing diagnostic facts. Do not volunteer findings for steps they skipped; missed signs are marked, not rescued (do not prompt or hint, beyond a single neutral "is there anything else you wish to examine?" when they say they are done). When they finish (or the exchange clearly reaches examination's end), ask them to present their findings and diagnosis. Then run the viva from the case file's questions, adapting to what they missed. Keep each of your turns short and examiner-like.
- COMMUNICATION encounter (skills C┬ĘE┬ĘF┬ĘG): play the character in the case file's surrogate brief ŌĆö their knowledge, emotions, and behavioural rules (including how they respond to a good vs poor approach). Stay strictly in character; reveal facts only as the brief allows and only when the candidate's approach earns them. Press with the character's scripted questions and concerns at natural moments. Do NOT slip into examiner voice until the candidate says they are finished (or clearly closes the conversation) ŌĆö then debrief.
- CONSULTATION encounter (all 7 skills AŌĆōG): play the patient from the brief for the history; answer only from the brief's facts ("I'm not sure, doctor" for anything not covered). When the candidate examines, reveal findings per the examination rules above. After their presentation, switch to examiner Q&A from the case file.

GOLDEN RULES:
1. NEVER reveal, hint at, or confirm the diagnosis, the answer key, the model presentation, or model viva answers until the candidate has committed to a presentation and diagnosis (or explicitly ends the encounter). This includes indirect leaks: over-specific vocabulary, leading questions, or confirming a guess mid-encounter.
2. Use only the facts in the encounter brief. Never invent new clinical facts, results, or history. Patients don't know medical terminology they wouldn't plausibly know.
3. Findings are earned: reveal strictly manoeuvre-by-manoeuvre / question-by-question.
4. Once the candidate has presented and the viva is done (or they give up), switch to TUTOR DEBRIEF: state the diagnosis and key discriminators, compare their presentation with the model presentation, list what they found, missed, or invented, and give focused teaching from the viva material and grounding notes.

7. If the candidate types "[BEGIN ENCOUNTER]", open the encounter in role: examinations ŌåÆ invite them to begin examining; communication/consultation ŌåÆ open with the scene and, where the brief scripts one, the character's opening line.
```

## Managed access (Phase 1 ŌĆö email OTP + domain tiers + per-user metering)

Purpose: let anyone with an eligible email practise on the **server-held, metered** gateway key ŌĆö login-gated, domain-tiered, USD-capped per user ŌĆö while BYOK stays the free, no-login door. Binding behaviour spec: `../_MANAGED_TIER_PLAN.md` (decisions locked with Arthur 2026-07-09). Implementation: `lib/tiers.ts` (client-safe tier contract + `ManagedStatus`), `lib/managed.ts` (server core), `lib/pricing.ts` (server-side price table + cost math), `lib/db.ts` + `db/schema.sql` + `scripts/migrate.mjs` (Postgres), `app/api/auth/{request,verify,signout,status}/route.ts`. New dependency: `pg` (`nodemailer` carries over).

**The two doors ŌĆö and no third.** BYOK: free, no login, the key lives only in the browser and rides each request; zero operator cost or liability; its precedence is absolute (a request carrying `API_KEY_HEADER` never touches the server key or the meter). MANAGED: the server-held Vercel AI Gateway key behind email-OTP login, domain tiers, and per-user metering. The failure mode this design exists to prevent is the third, implicit door ŌĆö **anonymous access to the server key**. That is an open "drain me" endpoint: a bot maximises call COUNT regardless of per-call cost (cheap DeepSeek makes a low cap drain *faster*, not safer), and the examiner doubles as a free general-purpose tutor bot without even a jailbreak. Rate limits, CAPTCHAs, and daily caps only slow that down; a login gate stops it. Hence invariant 9: the server key is never reachable without a valid session whose live DB tier covers the model and whose reservation succeeded.

**Enablement (fail closed):** `managedEnabled()` requires ALL of `AUTH_SECRET`, the database (`dbConfigured()` Ōćö `DATABASE_URL`), and `DEMO_GATEWAY_API_KEY` (env name kept from Phase 0 ŌĆö already set on prod). Anything less: every managed endpoint answers generically and the app stays BYOK-only ŌĆö no response may reveal WHICH precondition is missing.

**Tiers** (`lib/tiers.ts` fixes what a tier MEANS; the DATABASE decides who is IN one):

| Tier | Eligibility | `TIER_MODELS` | `TIER_ALLOWANCE_USD` |
|---|---|---|---|
| `public` | major consumer email domains (gmail.com, outlook.com, yahoo.com, icloud.com, ŌĆ” ŌĆö a curated ALLOW-list, which blocks disposable-domain farming better than any deny-list) | `deepseek/deepseek-v4-pro` only | US$1 / calendar month |
| `institutional` | approved SG-healthcare domains (mohh.com.sg, moh.gov.sg, the SingHealth / NUHS / NHG cluster domains, ŌĆ” ŌĆö receiving the OTP at the domain IS the proof of affiliation) | `anthropic/claude-sonnet-4.6` + `deepseek/deepseek-v4-pro` | US$2 / calendar month |

Membership lives in Postgres ŌĆö `allowed_domains` (domain ŌåÆ tier) and `email_overrides` (per-address grants, optionally with a custom `monthly_allowance_usd`; overrides win over domain rows) ŌĆö and is owner-editable with plain SQL, no redeploy. `resolveTier()` re-derives the grant from those tables at OTP request, OTP verify, AND every examiner call, so deleting a row revokes outstanding sessions immediately (the old whitelist-re-check property, kept). Domain ELIGIBILITY is public product behaviour (responses may say "this domain is BYOK-only" ŌĆö that leaks nothing personal); whether an individual ACCOUNT exists is NOT public and must never be inferable from any response.

**OTP rules** (why a code and not a magic link: MOH-class mail-security gateways rewrite or strip links, silently breaking link sign-in for exactly the institutional users this tier serves; a 6-digit code is plain body text and survives):
- 6 digits (`crypto.randomInt`), stored ONLY as `HMAC-SHA256(AUTH_SECRET, email:code)` in `otp_codes` ŌĆö a leaked DB row cannot be brute-forced offline without the signing secret.
- Single-use (a conditional consume ŌĆö exactly one concurrent verifier can win), 10-minute expiry (`OTP_TTL_MS`), Ōēż5 verify attempts per code (`OTP_MAX_ATTEMPTS`) ŌĆö the attempt is burned atomically BEFORE the hash comparison, so 10^6 combinations cannot be walked. Attempts and expiry live in Postgres ŌĆö durable across serverless cold starts (the old in-memory limiter could be reset by one).
- Exactly one live code per email: requesting a new code consumes every prior unconsumed one.
- Send-rate: 5/hour per email (`OTP_EMAIL_SEND_LIMIT` ŌĆö DURABLE, counted from `otp_codes` rows) + 30/hour per IP (`OTP_IP_SEND_LIMIT` ŌĆö in-memory best-effort, deliberately looser: hospital NAT puts many users behind one address, and `x-forwarded-for` is only meaningful behind a trusted proxy such as Vercel).
- Delivery (`sendOtpEmail`, never throws): dev without SMTP ŌåÆ the code is logged to the server console (the whole flow is testable with zero mail config); production with missing/partial SMTP ŌåÆ a loud error WITHOUT the code (a live credential must never land in deployment logs). Codes carry no links, so `APP_BASE_URL` is no longer sign-in-critical (it remains available for any absolute URL the server builds).

**Session** (`MANAGED_SESSION_COOKIE` = `paces_session`): a stateless HMAC token `{sub: users.id, email, purpose: 'managed_session', exp}` signed with `AUTH_SECRET`, in an httpOnly cookie (Secure in production, SameSite=Lax), TTL 30 days (`SESSION_TTL_MS`). The purpose string is disjoint from the retired demo purposes, so no legacy `demo_session` or link token can ever validate here. **The token is identity only ŌĆö authorization is re-derived from the database on EVERY use** (`/api/auth/status` and every examiner call re-run `resolveTier`), so allow-list edits apply to outstanding sessions immediately; nothing tier- or allowance-shaped is ever trusted from the cookie.

**Metering ŌĆö reserve-then-settle** (`reserveSpend`/`settleSpend` in `lib/managed.ts` + `lib/pricing.ts`; an LLM call's cost is only known after the reply):
1. **Reserve (atomic pre-call gate):** `reserveSpend(userId, estUsd, allowanceUsd)` runs ONE transaction with two conditional UPDATEs ŌĆö the user's monthly meter (`user_balances`: `reserved_usd += est` only `WHERE allowance_usd ŌłÆ spent_usd ŌłÆ reserved_usd Ōēź est`; 0 rows ŌćÆ `'user_cap'`), then the global SGT-day backstop (`global_spend` checked against `MANAGED_DAILY_CAP_USD`, default $5; 0 rows ŌćÆ `'global_cap'`, and the transaction rollback releases the user reservation). The estimate (`estimateCallUsd`) is a REALISTIC upper bound, not a theoretical worst case: system blocks ~45k tokens at the cache-READ rate, transcript at chars/3.5 on the input rate, output at max_tokens ├Ś `reasoningFactor`, +25% headroom, floored at $0.005 ŌĆö an occasional first-call cache write can settle above its reserve, which simply blocks the NEXT call (accepted overshoot of at most one call).
2. **Settle (post-call):** release the reservation and move the ACTUAL cost into `spent_usd` plus one append-only `usage_events` row. Actual cost comes from OUR OWN price table (`actualCallUsd` over the response's real usage ŌĆö gateway zero-markup list prices, verified 2026-07-09), never from the gateway's per-generation cost reporting (Pro-only + lagging; periodic reconciliation use only). DeepSeek V4 Pro's always-on thinking bills as OUTPUT and is already inside `output_tokens`, so settle needs no special-casing (only the reserve estimate inflates by `reasoningFactor` 2.5); DeepSeek cache writes are charged at 1├Ś input, deliberately conservative ŌĆö the ledger must never under-charge. Settlement is IDEMPOTENT on the provider generation id (partial unique index on `usage_events.generation_id` ŌĆö a retried settle cannot double-charge); on upstream error settle with `usage = null` (reservation released, nothing charged). `settleSpend` never throws: after a successful upstream call a metering failure must not become a user-facing error ŌĆö it logs loudly, and the manually-topped-up gateway balance still bounds real spend.
3. **Ledger vs meter (never merge):** `usage_events` is the APPEND-ONLY audit ledger; `user_balances` is the MUTABLE per-(user, period) meter; `global_spend` is the per-day backstop row. Periods are calendar months in **Asia/Singapore** (`'YYYY-MM'`, `sgtMonth()`); the backstop keys SGT days (`'YYYY-MM-DD'`, `sgtDay()`). `allowance_usd` is refreshed from config on every ensure ŌĆö tier/override edits apply mid-month; `spent`/`reserved` are never touched by that refresh.

**API routes** (all `runtime = 'nodejs'`; every error is a FIXED generic string ŌĆö upstream/DB error text is never forwarded to the client):
1. `POST /api/auth/request` `{email}` ŌåÆ `AuthRequestResponse` (`lib/types.ts`): `'sent'` for ANY plausible address on an eligible domain ŌĆö whether an account exists is never revealed; `'byok_only'` when the domain is on neither allow-list (public product behaviour, safe to state). `requestOtp`'s `'rate_limited'` surfaces as a 429 with a fixed string; `'unavailable'` (door not configured) fails closed with a fixed generic failure that names no missing precondition.
2. `POST /api/auth/verify` `{email, code}` ŌåÆ on success sets `paces_session` and returns `AuthVerifyResponse` `{ok: true}`. EVERY failure ŌĆö malformed, wrong, expired, consumed, attempt-capped, domain-removed-since-request, door-unconfigured ŌĆö collapses to the same fixed generic rejection: no oracle for which check failed.
3. `POST /api/auth/signout` ŌåÆ clears the cookie.
4. `GET /api/auth/status` ŌåÆ `ManagedStatus` (`lib/tiers.ts`): `{active: false}` unless door enabled + valid cookie + live DB grant; otherwise the MASKED email (`maskEmail` ŌĆö the full address never leaves the server), `tier`, the tier-filtered `models`, `allowanceUsd`/`spentUsd`/`reservedUsd`, and the current `period`.
5. `POST /api/examiner` integration: a BYOK `API_KEY_HEADER` takes precedence absolutely (handled exactly as before; never touches the server key or the meter). Otherwise the managed path requires, in order: a valid `paces_session`; a live `resolveTier` grant; the requested model Ōłł `TIER_MODELS[tier]` AND `isMeteredModel`; `reserveSpend` returning `'ok'`. Only then does `managedGatewayKey()` go straight into the SDK constructor (invariant 2 handling); after the call, `settleSpend` with real usage (or `null` on upstream failure). Failures map to fixed strings: no session ŌåÆ the existing keyless 401 (surfaced client-side as sign-in guidance); a session whose tier doesn't cover the model ŌåÆ 403 with switch-model/BYOK guidance; `'user_cap'`/`'global_cap'` ŌåÆ a fixed allowance-exhausted message that distinguishes only "your monthly allowance" vs "the app's daily budget", never numbers or SQL detail.

**Invariants (violations are critical bugs, same standing as the list at the top ŌĆö 9 and 10 there are the load-bearing ones)**
- The DB allow-lists are enforced at OTP request, OTP verify, AND every use of the session (status + examiner) ŌĆö removing an `allowed_domains`/`email_overrides` row revokes outstanding codes and cookies immediately.
- The server gateway key goes from `process.env` straight into the SDK constructor, only behind a valid session + live tier + successful reservation (invariants 2, 6, 9); it is never stored, logged, echoed, or client-readable in any form.
- BYOK precedence is absolute: a request carrying `API_KEY_HEADER` never touches the server key or the meter.
- No managed endpoint may reveal server configuration beyond the documented public fact of domain eligibility; account existence, verify-failure reasons, and missing-precondition detail must never be inferable from status code, body, or timing.
- Metering is server-truth (invariant 10): no client input can alter what is reserved or charged, and the ledger is append-only.
- The managed gate changes nothing else: answer-key hiding (invariant 1), `PublicCaseMeta` projection, model allowlist, path-traversal rules, and all other existing invariants hold unchanged with or without a session.

**Env** (documented in `.env.example` / README): `DATABASE_URL` (Neon Postgres from the Vercel Marketplace integration, or any plain Postgres; schema applied by the idempotent `scripts/migrate.mjs` ŌĆö tables `users`, `allowed_domains`, `email_overrides`, `otp_codes`, `usage_events`, `user_balances`, `global_spend`), `AUTH_SECRET`, `DEMO_GATEWAY_API_KEY`, `MANAGED_DAILY_CAP_USD` (default 5), `SMTP_HOST`/`SMTP_PORT`/`SMTP_USER`/`SMTP_PASS`, and the dev-only `MANAGED_LOG_OTP=1` (prints the plaintext code to the console when SMTP is unset AND `NODE_ENV !== 'production'` ŌĆö absent by default so a live code is never logged unless deliberately opted in). WHO may sign in is database config (`allowed_domains` + `email_overrides`), not env.

**Metering is a HARD bound (review 2026-07-09).** `estimateCallUsd` (`lib/pricing.ts`) is a genuine upper bound on one request's real spend ŌĆö system blocks priced at the cache-WRITE rate for every call, output at `max_tokens ├Ś reasoningFactor`, all ├Ś the max upstream call count (chat = 1 + `MAX_TOOL_ITERATIONS`, mark = 1). So the per-user allowance and the global daily cap actually bound settled spend, not merely the reserved estimate; `settleSpend` immediately releases the reservation down to the real `actualCallUsd`, so a single serialized user barely feels the over-reservation. The reservation carries the SGT `period`/`day` it was booked against and `settleSpend` is handed those exact keys (never recomputed), so a call crossing an SGT midnight/month boundary still settles into the row it reserved. Concurrency is row-lock-serialized (the conditional `UPDATE`s target one balance/global row), and OTP send-rate is enforced under a per-email `pg_advisory_xact_lock` so bursts can't over-send.

**Accepted residual risks (Phase 1 ŌĆö documented, not bugs):** (1) the 30-day `paces_session` is stateless, so sign-out clears only that browser's cookie ŌĆö a stolen cookie is bounded by the per-user monthly cap (the plan's designed defence) and by allow-list revocation (tier is re-checked live every call), not by server-side session kill; a future `session_epoch` column would add per-user revocation. (2) `settleSpend` never throws ŌĆö on a DB failure after a delivered reply it logs and makes a best-effort reservation release; if that also fails, the reservation self-clears at period reset and the gateway prepaid balance is the ultimate hard ceiling. (3) `usage_events` idempotency keys on the gateway `generation_id`; every current metered provider returns one, so the null-id path is latent only.
